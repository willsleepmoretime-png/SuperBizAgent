# 任务二：记忆与上下文优化

## 1. 这次要解决什么问题

当前项目已经有简单的对话记忆，但记忆保存在 `ChatController` 的内存 Map 中。

这会带来几个问题：

- 服务重启后，对话记录会丢失。
- 多个服务实例之间不能共享对话记录。
- 当前主要根据 `sessionId` 区分会话，用户隔离不够明确。
- 历史对话直接拼接到 Prompt，内容多了以后容易超过 Token 限制。
- 目前没有真正的长期记忆，Agent 无法复用过去已经验证过的排障经验。

本任务的目标是：

1. 用 Redis 保存短期对话记忆。
2. 用 PostgreSQL 保存已经验证的长期记忆。
3. 控制进入 Prompt 的记忆数量，避免上下文过长。
4. 让用户能够查看、清空和删除自己的记忆。

## 2. 当前代码情况

项目中已经存在以下能力：

| 代码位置 | 当前能力 | 存在的问题 |
| --- | --- | --- |
| `ChatController` | 使用 Map 保存会话 | 重启后丢失，Controller 职责太多 |
| `SessionInfo` | 保存最近 6 轮对话 | 只能在当前 JVM 中使用 |
| `SessionInfo` | 旧对话生成滚动摘要 | 只是简单截断和拼接，不是真正的摘要 |
| `application.yml` | 配置窗口大小和摘要长度 | 配置可以继续使用，但需要重新整理 |
| `SingleAgentPromptBuilder` | 把摘要和历史对话加入 Prompt | 没有统一 Token 限制 |
| `AgentRequest` | 已经有 `sessionId` 和 `userId` | `userId` 目前没有被严格使用 |
| `IncidentState` | 保存 AIOps 当前任务状态 | 任务状态只在内存中，不能恢复 |

因此，这次不是完全重写，而是把已有的内存记忆迁移到 Redis，再增加长期记忆。

## 3. 简化后的记忆结构

### 3.1 短期记忆

短期记忆保存当前会话最近的内容，例如：

- 最近 6 轮用户和助手对话。
- 较早对话的摘要。
- 当前正在讨论的服务和环境。

短期记忆使用 Redis 保存。

```text
memory:conversation:{userId}:{sessionId}
```

示例：

```json
{
  "recentMessages": [],
  "summary": "用户正在排查 payment-service 的 CPU 告警",
  "activeEntities": {
    "service": "payment-service",
    "environment": "production"
  },
  "updatedAt": "2026-07-11T12:00:00+08:00"
}
```

默认规则：

- 保留最近 6 轮完整对话。
- Redis 数据保留 7 天。
- 每次继续对话时刷新过期时间。
- 超出 6 轮的内容生成摘要。
- 用户清空会话时直接删除对应 Redis Key。

### 3.2 长期记忆

长期记忆保存以后可能再次使用的信息，例如：

- 已经确认的事故原因。
- 已验证有效的解决方案。
- 服务与数据库之间的依赖关系。
- 用户明确表达的稳定偏好。

长期记忆使用 PostgreSQL 保存，并使用 pgvector 支持相似内容检索。

### 3.3 AIOps 任务状态

AIOps 的执行状态与聊天记忆不是一回事，应该使用独立 Redis Key：

```text
aiops:task:{taskId}
```

它主要保存：

- 当前执行到第几轮。
- 已经执行了哪些步骤。
- 已经收集到哪些 Evidence。
- 任务是否完成或失败。

任务结束后，可以从这些数据中提取长期记忆候选，但不能把整个任务状态直接当成长久记忆。

## 4. 记忆如何进入 Prompt

当前代码会把历史对话直接拼接到系统 Prompt。改造后统一按下面的顺序加入上下文：

```text
系统提示词
→ 用户当前问题
→ 最近对话
→ 相关实体信息
→ 已验证的长期记忆
→ RAG 检索文档
```

基本规则：

- 用户当前问题必须完整保留。
- 最近对话比旧摘要更重要。
- 长期记忆默认只取最相关的 3 条。
- RAG 文档默认取最相关的 3 条。
- 超过 Token 限制时，优先删除相关度较低的旧记忆。
- 历史记忆只能作为参考，当前事故仍然要通过工具重新确认。

建议增加一个统一对象：

```java
public record MemoryContext(
    String summary,
    List<MemoryMessage> recentMessages,
    List<MemoryItem> entityMemories,
    List<MemoryItem> longTermMemories,
    int estimatedTokens
) {}
```

以后 `SingleAgentPromptBuilder` 和 `PlannerService` 都使用 `MemoryContext`，避免各自读取和拼接记忆。

## 5. 长期记忆的写入规则

Agent 不能把自己的所有回答都直接保存为长期记忆。

推荐流程：

```text
对话或任务结束
→ 提取可能有价值的内容
→ 保存为候选记忆
→ 人工或系统确认
→ 变成正式长期记忆
```

可以保存为候选的内容：

- 带 Evidence ID 的事故结论。
- 工具确认过的服务关系。
- 用户明确表达的长期偏好。
- 已经执行并确认有效的解决方案。

不能直接保存的内容：

- Agent 猜测的事故原因。
- 查询失败以后做出的推断。
- 大段原始日志。
- 密钥、Token、Cookie 等敏感信息。
- 已经保存在 Milvus 中的企业文档全文。

长期记忆可以使用这些状态：

```text
CANDIDATE          候选
HUMAN_VERIFIED     人工确认
SYSTEM_VERIFIED    系统确认
REJECTED           已拒绝
EXPIRED            已失效
```

默认只有人工确认和系统确认的记忆可以进入 Prompt。

## 6. 长期记忆表建议

第一版不需要设计得过于复杂，先建立一张核心表：

```sql
CREATE TABLE long_term_memory (
    memory_id UUID PRIMARY KEY,
    tenant_id VARCHAR(64),
    user_id VARCHAR(64) NOT NULL,
    memory_type VARCHAR(32) NOT NULL,
    content TEXT NOT NULL,
    summary TEXT,
    source_type VARCHAR(32) NOT NULL,
    source_id VARCHAR(128),
    evidence_ids JSONB,
    confidence NUMERIC(5,4),
    verification_status VARCHAR(32) NOT NULL,
    embedding vector(1024),
    created_at TIMESTAMPTZ NOT NULL,
    updated_at TIMESTAMPTZ NOT NULL
);
```

说明：

- `user_id` 用于用户隔离。
- `source_id` 可以保存 `taskId` 或 `sessionId`。
- `evidence_ids` 保存结论使用了哪些证据。
- `verification_status` 控制该记忆能否进入 Prompt。
- `embedding` 的维度最终要与实际使用的 Embedding 模型保持一致。

如果第一版时间有限，实体表和关系表可以放到第二阶段，不必一次全部完成。

## 7. 推荐代码结构

```text
org.example.memory
├── MemoryContext.java
├── MemoryRetriever.java
├── WorkingMemoryService.java
├── WorkingMemoryRepository.java
├── RedisWorkingMemoryRepository.java
├── LongTermMemoryService.java
├── LongTermMemoryRepository.java
├── ContextBudgetManager.java
└── MemoryController.java
```

主要职责：

- `WorkingMemoryService`：管理最近对话、摘要、TTL 和清空。
- `LongTermMemoryService`：管理候选记忆、确认、拒绝和删除。
- `MemoryRetriever`：读取短期和长期记忆。
- `ContextBudgetManager`：控制放入 Prompt 的内容长度。
- `MemoryController`：提供查看和删除记忆的接口。

## 8. 代码改造步骤

### M1：先整理用户和会话标识

- 明确 `AgentRequest.id` 表示 `sessionId`。
- 普通对话必须提供 `userId` 和 `sessionId`。
- 所有记忆都按 `userId + sessionId` 查找。
- 增加测试，保证不同用户不能读取同一个会话的内容。

### M2：把短期记忆迁移到 Redis

> 状态：已完成第一版（Redis 持久化、最近 6 轮、7 天 TTL、清空、普通/流式共用服务、故障降级和用户隔离测试）。

- 引入 Redis 依赖和配置。
- 新建 `WorkingMemoryRepository`。
- 把 `SessionInfo` 从 `ChatController` 中移出。
- `/chat` 和 `/chat_stream` 使用同一个记忆服务。
- 实现最近 6 轮、7 天 TTL 和清空接口。
- Redis 不可用时允许无记忆回答，并记录错误日志。

### M3：统一上下文和 Token 限制

> 状态：已完成第一版（统一 `MemoryContext`、最近完整问答对裁剪、摘要预算、普通响应与 SSE 的 `ContextUsage`）。长期记忆与 RAG 预算将在对应模块接入后补充。

- 新建 `MemoryContext`。
- 新建 `ContextBudgetManager`。
- 修改 `SingleAgentPromptBuilder`，不再直接接收历史 Map。
- Planner 也通过 `MemoryContext` 读取历史信息。
- 在响应中展示本轮使用了多少短期记忆和长期记忆。

### M4：增加 PostgreSQL 长期记忆

- 引入 PostgreSQL 和数据库迁移工具。
- 创建 `long_term_memory` 表。
- 引入 pgvector。
- 实现按 `userId` 过滤和向量相似度查询。
- 长期记忆默认只返回最相关的 3 条。

### M5：增加候选确认功能

- 对话或 AIOps 任务结束后提取候选记忆。
- 候选记忆不能直接进入 Prompt。
- 提供确认、拒绝和删除接口。
- 保存对应的 `taskId` 和 Evidence ID。

### M6：前端展示

- 展示本轮使用了哪些记忆。
- 展示记忆来源、时间和确认状态。
- 支持清空当前会话。
- 支持删除长期记忆。
- 支持确认或拒绝候选记忆。

## 10. 验收标准

- 服务重启后，用户仍然能恢复最近对话。
- 相同用户和会话能够读取自己的历史。
- 不同用户不能读取彼此的记忆。
- Redis 中的会话能够自动过期。
- 清空会话后，旧历史不会再次进入 Prompt。
- Prompt 总长度不会超过设置的 Token 预算。
- 未确认的长期记忆不会被默认使用。
- AIOps 的历史结论不会代替当前工具证据。
- 用户可以查看和删除自己的长期记忆。
- Redis 或 PostgreSQL 暂时不可用时，普通问答仍然可以降级执行。

## 11. 推荐实施顺序

建议按以下顺序开发：

```text
用户隔离
→ Redis 短期记忆
→ MemoryContext 和 Token 控制
→ PostgreSQL 长期记忆
→ 候选确认
→ 前端展示
```

第一阶段先完成用户隔离和 Redis。因为当前最直接的问题是会话重启后丢失，以及不同用户之间缺少清晰隔离。

pgvector 和实体关系可以稍后开发。它们有展示价值，但不是当前最优先的问题。

## 12. 我的看法

这个项目不需要一开始就做成非常复杂的企业记忆平台。

比较合适的做法是：

1. 先把当前内存中的 6 轮对话迁移到 Redis。
2. 再用一个简单的 `MemoryContext` 控制 Prompt 内容。
3. 最后增加 PostgreSQL 长期记忆和候选确认。

这样每一步都可以单独运行、测试和演示，也更适合写进简历。

项目真正值得展示的不是“用了多少种数据库”，而是：

- 对话重启后不会丢失。
- 不同用户的数据不会混在一起。
- 未确认的错误结论不会反复影响 Agent。
- 上下文长度和 Token 成本可以控制。

## 13. 简历可展示点

- 使用 Redis 实现可恢复的短期对话记忆和自动过期。
- 使用 PostgreSQL/pgvector 保存并检索已经验证的长期记忆。
- 通过候选、确认和失效状态避免错误记忆污染后续回答。
- 使用统一 `MemoryContext` 控制 Single Agent、AIOps 和 RAG 的上下文长度。
